Briefly summarize The Gaming Room client and their software requirements. Who was the client? What type of software did they want you to design?

In this project the client was The Gaming Room, and they developed a game called Draw It or Lose It for the Android OS. The Gaming Room wanted their game to be 
redesigned as a web based application so that it could run on multiple platforms and expand its customer base. The objective of this project is to allow multiple 
teams and players to join and play the game simultaneously. Also the system must ensure that one game instance is running in memory at a time and that teams and 
player names are unique. 

What did you do particularly well in developing this documentation?

I think I did a good job clearly explaining how the system would work and organizing the requirements in a way that makes sense. I broke down the relationships 
between the Game, Team, and Player classes and explained how they connect through the Entity base class. I also clearly explained the platform recommendations 
and why Linux would be a good choice. Overall, I feel like the document shows that I understood both what the client wanted and how to design a system to meet 
those needs.


What about the process of working through a design document did you find helpful when developing the code?

Working through the design document helped me plan everything before jumping into the code. It made me think about how the classes would connect, what each part 
of the system was responsible for, and how to avoid problems like duplicate names or multiple game instances. Having it mapped out first made the coding process 
less confusing and more organized. It basically gave me a clear roadmap instead of just guessing while coding.

If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could revise one part of my work, I would probably improve the security and scalability sections. I think I explained them clearly, but I could have gone
into more detail about how the system would handle a large number of users or protect user data. I would add more specific examples and maybe explain the technical 
side a little deeper to make it stronger and more professional.

How did you interpret the user’s needs and implement them into your software design? Why is it so important to consider the user’s needs when designing?

I interpreted the user’s needs by looking at what The Gaming Room specifically asked for and turning those into technical features in the design. For example, 
they needed unique game and team names, so I made sure the system included unique identifiers and checks to prevent duplicates. They also wanted only one game 
instance running at a time, so that was built into the design as well. It’s important to consider the user’s needs because the whole point of software is to 
solve their problem. If you don’t design around what they actually want, the system might work technically, but it won’t be useful to them.

How did you approach designing software? What techniques or strategies would you use in the future to analyze and design a similar software application?

I approached designing the software by first breaking down the requirements and figuring out what the system actually needed to do. Then I organized the main 
parts of the program into classes like Game, Team, and Player and made sure each one had a clear role. I focused on keeping the design simple, organized, and 
easy to expand later. In the future, I would continue starting with requirement analysis and using things like UML diagrams and object-oriented design principles 
to map everything out before coding. That way, the structure is clear from the beginning and it makes development smoother.
